# NestJS Backend Development Skill

## Core Principles

- **Type Safety**: TypeScript strict mode
- **Modular Architecture**: Feature-based modules
- **Dependency Injection**: NestJS DI only
- **Fail Fast**: Validate early, meaningful exceptions
- **Documentation**: Swagger on every endpoint
- **Middleware**: Express middleware via @nestjs/platform-express
- **ORM**: Prisma for type-safe database access

## Naming Conventions

### Casing Quick Reference

| What | Casing | Example |
|------|--------|---------|
| File names | kebab-case | `create-user.dto.ts` |
| Directories (domain) | kebab-case, singular | `user/`, `order-item/` |
| Directories (shared) | kebab-case, plural | `guards/`, `pipes/` |
| Classes | PascalCase + suffix | `UsersController`, `CreateUserDto` |
| Interfaces | PascalCase (no `I` prefix) | `JwtPayload` |
| Enums | PascalCase | `UserRole` |
| Enum members | UPPER_SNAKE_CASE | `ADMIN` |
| Enum values (string) | lowercase | `'admin'` |
| Constants | UPPER_SNAKE_CASE | `MAX_RETRY_ATTEMPTS` |
| Environment vars | UPPER_SNAKE_CASE | `DATABASE_URL` |
| Methods / functions | camelCase | `findOneByEmail()` |
| Variables / properties | camelCase | `firstName` |
| Booleans | camelCase with prefix | `isActive`, `hasRole`, `canDelete` |
| Route paths | kebab-case, plural nouns | `/user-profiles` |
| Entity classes | PascalCase, singular | `User` |
| DB table names | snake_case, plural | `users`, `order_items` |
| DB column names | snake_case | `first_name` |
| Event names (emitter) | dot.notation | `'user.created'` |
| Test files (unit) | `*.spec.ts` | `users.service.spec.ts` |
| Test files (e2e) | `*.e2e-spec.ts` | `users.e2e-spec.ts` |

### File Naming

Pattern: `<name>.<type>.ts` — kebab-case, dot-separated by type.

| Type | Pattern | Example |
|------|---------|---------|
| Controller | `<name>.controller.ts` | `users.controller.ts` |
| Service | `<name>.service.ts` | `users.service.ts` |
| Module | `<name>.module.ts` | `users.module.ts` |
| Guard | `<name>.guard.ts` | `jwt-auth.guard.ts` |
| Interceptor | `<name>.interceptor.ts` | `logging.interceptor.ts` |
| Filter | `<name>.filter.ts` | `http-exception.filter.ts` |
| Pipe | `<name>.pipe.ts` | `validation.pipe.ts` |
| Middleware | `<name>.middleware.ts` | `logger.middleware.ts` |
| Decorator | `<name>.decorator.ts` | `current-user.decorator.ts` |
| DTO | `<action>-<entity>.dto.ts` | `create-user.dto.ts` |
| Entity | `<name>.entity.ts` | `user.entity.ts` |
| Interface | `<name>.interface.ts` | `user.interface.ts` |
| Enum | `<name>.enum.ts` | `user-role.enum.ts` |
| Constant | `<name>.constant.ts` | `app.constant.ts` |

### Class Naming

PascalCase with a type suffix. Controllers and services use **plural** names (they manage collections); entities use **singular** (they represent one instance).

| Type | Convention | Example |
|------|-----------|---------|
| Controller | `<Plural>Controller` | `UsersController` |
| Service | `<Plural>Service` | `UsersService` |
| Module | `<Plural>Module` | `UsersModule` |
| Guard | `<Name>Guard` | `JwtAuthGuard` |
| Interceptor | `<Name>Interceptor` | `LoggingInterceptor` |
| Filter | `<Name>Filter` | `HttpExceptionFilter` |
| Pipe | `<Name>Pipe` | `ValidationPipe` |
| DTO | `<Action><Entity>Dto` | `CreateUserDto`, `UpdateOrderDto` |
| Entity | `<Singular>` | `User`, `ProductCategory` |
| Event | `<Entity><PastVerb>Event` | `UserCreatedEvent`, `OrderPlacedEvent` |

### Method Naming

camelCase. Do not duplicate class context — in `UsersService`, use `findOne()` not `findOneUser()`.

**Standard CRUD methods** (generated by `nest g resource`):

| HTTP | Route | Controller | Service |
|------|-------|-----------|---------|
| POST / | Create | `create()` | `create()` |
| GET / | List | `findAll()` | `findAll()` |
| GET /:id | Read | `findOne()` | `findOne()` |
| PATCH /:id | Update | `update()` | `update()` |
| DELETE /:id | Remove | `remove()` | `remove()` |

**Extended patterns**: `findOneByEmail()`, `findOneOrFail()`, `createMany()`, `countAll()`

### Route Paths

- Plural nouns, kebab-case, lowercase: `/users`, `/product-categories`
- No verbs in paths — the HTTP method conveys the action
- Nested resources for relationships: `/users/:userId/orders`
- Controller decorator: `@Controller('users')`

### Constants and Enums

```typescript

### Events

- **Event emitter names**: dot-notation past tense — `'user.created'`, `'order.placed'`
- **Event payload classes**: `<Entity><PastVerb>Event` — `UserCreatedEvent`
- **CQRS commands**: imperative verb first — `CreateUserCommand`
- **CQRS events**: noun first, past tense — `UserCreatedEvent`

## Code Standards

### Controllers
- @ApiTags, @ApiOperation, @ApiResponse decorators
- Validation pipes on all inputs
- Thin controllers — delegate to services

### Services
- Logger: `private readonly logger = new Logger(ServiceName.name)`
- Throw NestJS exceptions (NotFoundException, ConflictException, etc.)
- Transactions for multi-entity operations
- Follow structured logging standards (see Logging section below)

### DTOs
- class-validator on all fields
- @Transform for sanitization
- PartialType/OmitType for updates
- @Expose() in response DTOs

### Entities
- UUID primary keys
- @Index on query columns
- Soft deletes with @DeleteDateColumn

### Middleware
- Use Express middleware adapters via `@nestjs/platform-express`
- Apply middleware in modules or globally in main.ts
- Type middleware properly: `(req: Request, res: Response, next: NextFunction) => void`
- Common use cases: logging, request tracking, body parsing

### Database (Prisma ORM)
- **Schema**: Define in `prisma/schema.prisma`
- **Client**: Inject PrismaService via DI
- **Queries**: Use Prisma Client methods (findUnique, findMany, create, update, etc.)
- **Transactions**: Use `prisma.$transaction()` for multi-model operations
- **Relations**: Define relations in schema, use `include` for eager loading
- **Migrations**: Run `prisma migrate dev` for development, `prisma migrate deploy` for production
- **Type Safety**: Prisma generates TypeScript types automatically

### Logging

**Standard Log Structure** (consistent across all log levels):

```typescript
interface LogContext {
  // Required fields
  action: string;              // Operation identifier (e.g., 'createUser', 'processPayment')
  service: string;             // Service/class name

  // Optional but recommended
  userId?: string;             // User performing the action
  resourceId?: string;         // Primary resource being acted upon
  traceId?: string;            // Request correlation ID

  // Conditional fields
  payload?: Record<string, any>;   // Input data (sanitize sensitive fields)
  response?: Record<string, any>;  // Output data (sanitize sensitive fields)
  duration?: number;               // Operation duration in ms

  // Error fields (for error/warn levels)
  error?: {
    message: string;
    code?: string;
    stack?: string;
  };

  // Additional context
  metadata?: Record<string, any>;  // Any extra context
}
```

**Usage Pattern**:

```typescript
export class UserService {
  private readonly logger = new Logger(UserService.name);

  async create(createUserDto: CreateUserDto) {
    const context: LogContext = {
      action: 'createUser',
      service: UserService.name,
      payload: { email: createUserDto.email }, // Exclude password
    };

    try {
      const user = await this.prisma.user.create({ data: createUserDto });

      this.logger.log('User created successfully', {
        ...context,
        userId: user.id,
        response: { id: user.id, email: user.email },
      });

      return user;
    } catch (error) {
      this.logger.error('Failed to create user', {
        ...context,
        error: {
          message: error.message,
          code: error.code,
          stack: error.stack,
        },
      });
      throw new InternalServerErrorException('User creation failed');
    }
  }

  async findOne(id: string) {
    const start = Date.now();
    const context: LogContext = {
      action: 'findUser',
      service: UserService.name,
      resourceId: id,
    };

    const user = await this.prisma.user.findUnique({ where: { id } });

    this.logger.log('User fetched', {
      ...context,
      duration: Date.now() - start,
      response: user ? { id: user.id } : null,
    });

    return user;
  }
}
```

**Log Levels**:
- `debug`: Development/debugging (include verbose payload/response)
- `log`: Normal operations (include key identifiers, optional payload/response)
- `warn`: Recoverable issues (include error context)
- `error`: Failures (always include error object with stack)

**Best Practices**:
- Always use the same field names for consistency across services
- Sanitize sensitive data (passwords, tokens, PII) from payload/response
- Include `traceId` from request headers for distributed tracing
- Log payload/response for critical operations (auth, payments, data mutations)
- Use `duration` for performance-sensitive operations
- Never log full objects - extract relevant fields only

**Sensitive Data Sanitization in Logs**:

CRITICAL: Never log sensitive information. Always sanitize before logging.

**Always Exclude from Logs**:
- `password`, `passwordHash`, `currentPassword`, `newPassword`
- `token`, `accessToken`, `refreshToken`, `apiKey`, `secret`
- `creditCard`, `cvv`, `cardNumber`, `ssn`, `taxId`
- `privateKey`, `secretKey`, `encryptionKey`
- Full request/response bodies without filtering

**Sanitization Strategies**:

```typescript
// ❌ BAD - Logs sensitive data
this.logger.log('User login', {
  action: 'login',
  payload: loginDto, // Contains password!
});

// ✅ GOOD - Sanitize payload
this.logger.log('User login', {
  action: 'login',
  payload: {
    email: loginDto.email,
    // password excluded
  },
});

// ✅ GOOD - Use sanitization helper
const sanitizeForLog = (obj: any, sensitiveKeys: string[]) => {
  const sanitized = { ...obj };
  sensitiveKeys.forEach(key => delete sanitized[key]);
  return sanitized;
};

this.logger.log('User login', {
  action: 'login',
  payload: sanitizeForLog(loginDto, ['password']),
});

// ✅ GOOD - Mask partial data when needed
this.logger.log('Payment processed', {
  action: 'processPayment',
  payload: {
    cardLast4: card.number.slice(-4),
    // Full card number excluded
  },
});
```

**Required Sanitization Checklist**:
- [ ] Remove passwords from authentication payloads
- [ ] Remove tokens from authorization headers
- [ ] Mask credit card numbers (show last 4 only if needed)
- [ ] Exclude API keys and secrets
- [ ] Filter PII based on compliance requirements (GDPR, HIPAA)
- [ ] Sanitize error messages that may contain sensitive data

## PrismaService Setup

```typescript
// src/prisma/prisma.service.ts
import { Injectable, OnModuleInit, OnModuleDestroy } from '@nestjs/common';
import { PrismaClient } from '@prisma/client';

@Injectable()
export class PrismaService extends PrismaClient implements OnModuleInit, OnModuleDestroy {
  async onModuleInit() {
    await this.$connect();
  }

  async onModuleDestroy() {
    await this.$disconnect();
  }
}
```

```typescript
// src/prisma/prisma.module.ts
import { Global, Module } from '@nestjs/common';
import { PrismaService } from './prisma.service';

@Global()
@Module({
  providers: [PrismaService],
  exports: [PrismaService],
})
export class PrismaModule {}
```

**Usage in Services:**
```typescript
import { Injectable } from '@nestjs/common';
import { PrismaService } from '../prisma/prisma.service';

@Injectable()
export class UserService {
  constructor(private readonly prisma: PrismaService) {}

  async findOne(id: string) {
    return this.prisma.user.findUnique({
      where: { id },
      include: { posts: true },
    });
  }

  async create(data: CreateUserDto) {
    return this.prisma.user.create({ data });
  }
}
